This playbook describes behaviour that is implemented and running, not intentions. Where the system refuses to do something — delete a department, cancel work already begun, show an employee the AI's assessment of them — the refusal is documented as a rule, because that is what it is.

1What this system is

A record of what every employee did, block by block, that managers can trust and employees can live with.

Ara Workbench asks each employee one question per block of working time — hourly by default, or every two or three hours where that is the honest cadence: what did you complete? Those answers roll up three ways off a single table — by person, by project, by department — so progress can be reported without anyone maintaining a second set of books.

Around that core sit four things: assignments (work handed out, so hours can be checked against intent), approvals (a Tech Lead signs off each day), analytics (a nightly rollup feeding the dashboards), and an AI analyser that asks, on a cadence Management sets, whether the hours logged actually advance the work assigned.

The design position

Two decisions shape everything else, and both are unusual enough to state plainly.

An entry records work already done

A task entry therefore has no status and no priority. Every row is, by definition, completed work. A column that can only ever hold one value is not information — it is a field that invites people to lie to it, and a denominator that quietly inflates. Status lives on the day (an approval state) and on assignments and project modules, which are genuinely work still ahead.

The work is the company's, not the person's

Removing a person, a team, a department or an hour must never remove the record of work that was done. Deleting an employee removes their login; their logged hours stay, still attributed to them by name. This was once a cascade, and deleting one employee silently erased months of delivery history.

Who uses it

Departments are data, not an enum. Adding a fifth is a row, not a release: its working hours, its grid columns and any extra form fields are all configuration. The four seeded departments are Tech Team, Digital Marketing, Social Media Management and Video Editing.

Isolation

A Tech Lead of Video Editing must never, under any code path, see a Digital Marketing employee's task. This is not a check that each endpoint remembers to make — it is a filter that is impossible to omit. See §4.

Config is validated at boot

Every variable in .env is checked by src/config/env.js when the process starts. A bad config refuses to start rather than failing three hours later inside a request handler. If the API will not boot, read the first lines of its output — it names the variable.

Health and prefixes

Liveness is GET /healthz. Every API route sits under /api/v1. There is no /api/health; looking for one is the usual first wrong turn.

Seeded accounts

The seed creates one Management account plus a populated demo company. Every seeded user shares the password Password@2026!; the Management account's password is set from your environment. All seeded users are flagged mustChangePassword, so the first sign-in goes straight to a forced password change — which is also the fastest way to confirm the auth chain works end to end.

Settings that are not in .env

Business policy lives in Admin › Settings and takes effect without a redeploy: reminder grace minutes, auto-approval hours, the backdating window, autosave debounce, whether daily submission is required, and how often the AI analyser runs. Secrets and infrastructure — the database URL, the JWT secret, the SMTP password, the Gemini key — are environment variables only, and are never rendered or returned by any endpoint. An admin UI that can rewrite the JWT secret is an admin UI that can compromise the whole system.

3A day in the system

The same day, from three seats.

The employee

  1. Opens My Task Sheet. The grid's columns come from their department's configuration — Tech sees 10:00–18:00, Video Editing sees its own shift, from the same screen.
  2. Each block, fills a cell: what did you complete? plus a project. If they have open assignments, a third question appears first — which task? — with an honest "Other work" option.
  3. Text autosaves after 1.2 seconds. A small indicator says saving / saved / unsaved — never a blocking spinner.
  4. Logging an hour late is allowed and flagged LATE. The system measures the behaviour rather than preventing the record.
  5. When every required hour is filled, Submit becomes available. Until then its tooltip says how many hours are outstanding.

The Tech Lead

  1. Approvals shows submitted sheets oldest first, with filled/expected hours and how long each has been waiting.
  2. Review deep-links to Monitor — the person's actual hourly sheet — because approving a number without reading the work is not review.
  3. Approve, or Return with a note. The note is a required field in the dialog, not a 422 after the fact.
  4. A lead may correct an entry directly. It is badged LEAD EDIT in the grid and recorded against the lead's name in the revision history. A lead's edit is never silent.
  5. AI Insights lists findings from the analyser, each expandable to the exact evidence the model was given.

Management

  1. Dashboard opens on today, not a 30-day range: per-department compliance and a list of names to chase. Everything else is behind "Detailed analytics".
  2. Administration covers departments, employees, teams, projects, the audit log and settings.
  3. Management has no department and therefore no task sheet — they do not log hours, and the nav hides it.
Two thresholds, deliberately different

Two hours past an unfilled slot earns a private nudge. Three hours puts your name on the Management dashboard. The gap exists so that a nudge stays a nudge.

What happens without anyone

Nine scheduled jobs keep the system honest when nobody is looking: hourly reminders with a three-way escalation, a lead digest twice a day, a management summary at 19:00, auto-approval of sheets nobody reviewed within 24 hours, the nightly rollup, the 180-day retention sweep, and the AI analyser. See §8.

4Access: roles, permissions, scope

Two independent questions — may this role do this kind of thing at all? and to which rows?

Permissions, not role checks

No controller, service or repository in this codebase may contain if (user.role === 'MANAGEMENT'). Capabilities are named resource:verb; roles are bundles of them; routes declare the capability they need. At boot, the app throws if any declared permission is assigned to no role — otherwise it is dead code.

An unresolvable scope returns a sentinel no id can equal, so it matches nothing rather than everything. A Tech Lead with no department is downgraded to SELF — deny, never silently escalate. A bug here leaks an empty page, not another department's data.

A requested filter is merged with the permitted scope, and the permitted scope always wins. An employee asking for someone else's rows is denied outright, not silently rewritten to their own — silently rewriting would mask a broken client.

The UI is not the guard

The frontend hides buttons the user cannot use, using permissions the server sent. Those are cosmetic affordances. The API authorises independently and returns 403. An unauthorised page shows an explanatory lock screen inside the normal app shell — not a redirect, not a blank.

5Module reference

Every backend module, what it is for, and the rules it enforces. The companion Flowcharts PDF has one diagram per module.

auth

Sign-in, sessions, password recovery, own profile. Access tokens are 15-minute JWTs; refresh tokens are opaque, stored only as SHA-256 hashes, and delivered in an httpOnly, secure, strict-sameSite cookie.

  • Rotation with reuse detection. Every refresh mints a successor and revokes its predecessor. Replaying a used token revokes the entire token family and bumps the password epoch, signing the account out everywhere.
  • Token epoch. Any access token issued before passwordChangedAt is rejected, so a password change, a demotion or a department move cannot be outlived by a stale token.
  • No enumeration. Forgot-password always answers identically. Login returns one generic error, and the inactive-account check happens after the password check.
  • The policy is not applied at login — applying it there tells an attacker which passwords could not exist.
  • Lockout after 5 failures for 15 minutes, held in the database so it survives scale-out.

users

Employee administration, scoped so a lead sees only their department.

  • You cannot change your own role or status, or delete your own account.
  • The last active Management account cannot be demoted, deactivated or deleted.
  • A Tech Lead who still leads an active team cannot be moved to another department — that would leave a team led by someone who cannot see it.
  • Changing role, department or status revokes every session immediately.
  • A department-scoped caller may reset employees' passwords only. Handing one lead the power to issue another lead a password is a lateral takeover, not a helpdesk favour.
  • Deletion stamps names onto history first, then deletes the account — after the delete the foreign key is already NULL and there is no way left to find which rows were theirs.

departments

Departments plus their working-hour columns and bespoke task fields.

The logging cadence is Management's to set

Each department carries a logging interval (30 minutes to 4 hours), a day start and a day end. Admin › Departments › Working hours regenerates that department's grid columns from them. "Hourly" was an assumption baked into the product; it is now a setting, because the honest cadence differs by the kind of work. An editor mid-render or an engineer mid-debug cannot describe a completed unit of work every sixty minutes, and forcing them to produces "continued task" in every row — which is worse than no data, because it looks like data.

The change is two steps by design: preview, then apply. The preview is computed by the same planner as the write, and names every column that would be affected. requiredSlotsPerDay follows the new column count automatically — leaving it at 7 after halving the columns would mark every employee permanently non-compliant, and nobody would connect it to this change.

What a rebuild does to existing work

No task entry is ever deleted by a rebuild. A column the new layout has no place for is retired — it leaves the grid, its entries stay readable, exportable and attributable. Only genuinely empty columns are deleted. Because the database allows exactly one column per start time per department, a column whose start time survives with a different span is re-spanned in place: no entry moves, but the label above it changes (10:00–11:00 becomes 10:00–12:00). The preview lists those separately, with their entry counts, so that is a decision rather than a discovery. Overtime columns are never touched.

  • Deletion is refused while the department holds employees, teams, real projects or logged hours (409, naming the blocker). To retire one with history, deactivate it.
  • Time slots may not overlap. Removing a slot that has hours behind it retires it — a hard delete would cascade real work away.
  • Overtime appends the next hour after the last column, and is excluded from required hours. An overtime column that counted toward the requirement would silently make overtime mandatory. It can be removed only while empty for everyone, and never past midnight.
  • A custom field's key is immutable once entries carry it; change the label instead, which is what users actually see. Field deletion is soft, so old values stay queryable.

projects

  • Every logged hour must name a project, so every department is seeded with an Internal / Non-project catch-all. It cannot be deleted and cannot leave ACTIVE — a mandatory field with no honest answer is how data goes bad.
  • A project with logged hours cannot be deleted, only archived; archiving removes it from the picker while every hour keeps its index.
  • A project cannot change department, or its hours would land on the wrong side of the isolation boundary.
  • The dropdown offers ACTIVE projects only — offering an archived one is how dirty data gets created.
  • Modules are the deliverables a project is made of. A module with assignments is retired, not deleted. completedAt is derived from the transition and cleared on reopen: a completion date on something still in progress is a lie the reports believe.

projects — who is on one

Until recently nobody belonged to a project: people belonged to departments and teams, projects belonged to departments, and the only trace of who worked on what was whoever had logged an hour. That answers a question about the past. It cannot say who is on a project before the first hour is logged, and it cannot be corrected when it is wrong.

Project membership now exists as its own record. The Assign task dialog reads department → project → person, and the people list puts members at the top with the hours they have logged, everyone else in the same department below them in muted text.

Muted, not hidden

Non-members are still selectable, deliberately. A brand-new project has no members at all, so a members-only list could never be given its first person — and putting somebody onto a project is one of the most ordinary reasons to open the dialog. Choosing one asks: "They are in the right department but not on this project. Assign the task and add them?" Answering yes does both, in a single transaction, so you can never end up with someone holding work on a project they are still not on.

Membership is current state, not history. It is the one place in this schema where deleting a person removes the row — the record of what they did lives on their task entries and assignments, which are preserved as always. Taking somebody off a project likewise removes nothing they did. Membership never crosses a department boundary.

assignments

Forward-looking work handed to a person. Hours logged against an assignment form its progress thread, and complete the chain the AI reasons over: hour → assignment → module → project.

  • States: ASSIGNED → IN_PROGRESS → SUBMITTED → DONE, with reopen as the only way back.
  • Cancellation is legal only from ASSIGNED. Once an hour has been logged, nobody — including the creator — can cancel it away: real effort is on the record.
  • There is no delete, ever.
  • Two-party handshake: only the assignee may submit; nobody may review their own work, including a Tech Lead who was assigned it.
  • The assignee and department are fixed at creation and taken from the assignee, never from the client. The project is immutable; only the module may be re-pointed, and only within the same project.
  • A percentage complete is shown only where an estimatedHours estimate exists. Otherwise the UI shows effort and never a guessed percentage.
  • The first hour logged flips ASSIGNED → IN_PROGRESS inside the same transaction that saves the entry.

tasks

The hourly grid and the day approval workflow. Five invariants: scope, locking, concurrency, history, backdating.

  • Day states: DRAFT → SUBMITTED → APPROVED or REJECTED. Editing a rejected sheet returns it to DRAFT automatically. An approved day can be reopened only by a lead, as an explicit audited override.
  • Two things lock a day: its status (SUBMITTED/APPROVED), and age beyond the backdating window (default 2 days). Future dates are refused outright.
  • Lateness is measured, not blocked. isLate is computed server-side from the slot's end plus grace — never trusted from the client, never inferred from updatedAt.
  • Submission requires completeness, counted against the real current slots rather than the configured integer, so an admin's grid change cannot demand a column that no longer exists.
  • Autosave never creates an unprojected row. It is skipped and reported as skipped, so the cell stays honestly "unsaved" — which is exactly what is true.
  • Every mutation writes a revision in the same transaction, storing the project name as well as its id so the history stays readable in two years after a rename. A delete writes its tombstone before the row goes.
  • Optimistic concurrency: a stale version returns 409 with the server's current row attached, so the UI can offer "use theirs / keep mine" instead of losing the edit.
  • Break slots are not writable. Overtime slots are — that is the entire point of them.

reports

  • Excel, CSV or PDF, streamed, always scoped to the caller — a lead's export contains only their department.
  • Every export writes a REPORT_EXPORTED audit row. Exporting data is an event a compliance officer will one day want to see.
  • CSV is escaped against formula injection.

dashboard

  • The compliance denominator counts only hours that have actually finished, excluding the in-progress hour, breaks and overtime. A metric that is red by construction every morning is one everybody ignores by lunchtime.
  • Team follow-up separates fill rate from on-time rate: a team at 100% fill and 30% on-time is backfilling, not complying.
  • Departments with zero activity are kept on the chart. A department that logged nothing is the most important bar on it.

notifications

  • You can only read and mark your own — there is no userId parameter anywhere in this module.
  • Persist first, email second. The bell is the system of record; email is best effort.
  • De-duplication is a UNIQUE index in the database, not an application check that races itself the moment there is more than one instance. Only the rows that actually landed are emailed.
  • Deactivated accounts are never mailed — that address may have been reassigned.

audit & settings

  • The audit log is append-only: no update or delete endpoint exists, and the retention job explicitly spares it. The record of who deleted the data must outlive the data.
  • Audit writes never break a user action — they are fire-and-forget and loudly logged on failure, except where atomicity matters (role changes, approvals, deletes), where they join the transaction.
  • Sensitive keys are dropped entirely, not masked, from audit diffs.
  • Settings accepts only known keys — an open key-value store just ships typos.

7The AI analyser

On the cadence Management sets — two hours by default — per person: do the hours logged plausibly advance the work assigned?

What it does

For each candidate it builds one evidence object — their open assignments, and the hour descriptions they wrote in the window — and asks Gemini for a structured verdict: a kind (MISALIGNED, IDLE, LOW_SUBSTANCE, AT_RISK, ON_TRACK), a severity, an optional 0–100 alignment score, a finding and a recommendation.

The four rules it enforces

  1. Write the finding before telling anyone. A model's judgement about a person's work is an accusation with consequences. It lands in a durable row — with the evidence it saw and the model that said it — so a disputed finding can be audited against its inputs instead of re-argued from memory.
  2. The employee is notified, not shown. Leads and Management get the finding. The employee gets a neutral, actionable nudge — never the reasoning, never a score. They can fix the record; they are not handed a machine's opinion of them. GET /ai/insights returns 403 for an employee, enforced in the service as well as the route.
  3. Evidence or silence. No assignments and nothing logged means nothing to judge. Absence of data is not evidence of idleness, and inventing a finding from it is how a tool like this loses its users' trust in a week.
  4. It never blocks anything. No timesheet is rejected, no account touched. Every failure path — no key, no quota, bad JSON — degrades to "no insights this run".

ON_TRACK findings are stored too. An insight table that only ever records problems reads as a charge sheet, and makes it impossible to tell "the AI checked and was satisfied" from "the AI never looked".

Per-department opt-out

Enforced in the SELECT, not as a filter on the results

Department.aiAnalysisEnabled is part of the analyser's candidate query. A department that has switched it off never becomes a candidate, so its assignments and hour descriptions are never read out of the database and nothing about it leaves the network. Analysing everyone and hiding the findings afterwards would mean the work had already left by the time we honoured the setting. That is not an opt-out.

The toggle is Management-only and audited. /ai/status reports which departments have opted out, so "nothing flagged" is never ambiguous.

What leaves the network

For an opted-in department only: employee name, code and department; the analysis window; their open assignments (title, description, project, module, status, priority, due date, estimate); and the hour descriptions they wrote, with the project and assignment each was logged against. No email addresses, no credentials, no salary or HR data — the system holds none.

The period review — the on-demand check

The scheduled analyser answers "is today going wrong?". It cannot answer "has this person moved in a fortnight?", because its window is two hours long. The period review does: Management picks a department and a period (7–30 days) on the AI Insights screen and presses Run AI detection.

Two further dropdowns — Project and Employee — narrow the question. Both are optional and both default to everything, so the button pressed as it stands is the whole-department review it has always been. A project gathers everyone connected to it three ways: on its member list, holding an assignment on it, or having logged an hour against it in the period. Membership alone would miss whoever is quietly helping out; hours alone would miss the person assigned to it who has not started, who is the more interesting of the two. Naming an employee overrides both — an explicit instruction is not a filter to be second-guessed.

Each active person in that department is read across the whole period at once — their daily log, their assignments, and crucially the status and completion date of the project modules that work belongs to. That last part is what turns a suspicion into a finding: hours logged against a module the project itself records as finished is a contradiction, not a similarity score.

What only a review can catch

Three entries — "tested communication module", "testing schoolmate communication module", "testing communication module of schoolmate app" — spread across a fortnight. Three different strings, so exact matching finds nothing, and each one read alone is unremarkable. Laid side by side against a module completed on day five, it is a finding: NO_PROGRESS.

Three guards that keep it honest

Repetition alone is never a finding. Testing one large module for three weeks is normal engineering. NO_PROGRESS requires restatement plus hard evidence — the module already complete, or effort past its estimate with nothing moving. A completed module alone is not enough either: bug fixes and follow-ups legitimately land after sign-off, so distinct advancing entries after completion read as ON_TRACK.

Absence of data can never be CRITICAL. This system holds no record of leave, sick days or onboarding, so a quiet fortnight is explained equally well by any of them. The cap is enforced in code, not in the prompt — the prompt already said it and the model returned CRITICAL anyway.

A project filter must not libel anyone. Narrowed to one project, the model sees that project's share of a person's time and nothing else — so somebody who spent the fortnight heads-down on a different project shows up with an empty log, which reads exactly like idleness. Withholding their other hours is right; withholding the fact of them is what turns a filter into an accusation. The evidence therefore carries one integer, coverage.entriesElsewhere, counting the hours held back. Two identical two-hour Schoolmate logs, one with 0 elsewhere and one with 14, correctly return NO_PROGRESS and ON_TRACK. Severity is capped at INFO in code for anyone who logged nothing here but plenty elsewhere: that is a fact about allocation, not about effort.

Nobody is notified by a review. The scheduled analyser alerts leads because it catches things worth correcting the same day; a review is retrospective, so the result goes to the person who asked and no further. Reviews are listed on their own tab, never mixed with the live feed — a fortnight-long judgement must not read as something that happened this afternoon. A project-scoped verdict is stored under its own key, so it sits beside the whole-period one rather than overwriting it; naming an employee does not, because reviewing one person over a period already reviewed is the same question asked again and should refresh the answer, not duplicate it.

Turning it off

Three levers, coarsest first: unset GEMINI_API_KEY (dormant); set AI_ANALYSIS_ENABLED=false (configured but off); or switch off individual departments in their admin drawer.

Why the locking matters

node-cron runs inside the application process, and App Service scales out. With three instances every expression fires three times: three reminder emails to each employee, and three concurrent 180-day delete transactions racing on the same rows. Each job takes a lease on a database row via a single atomic conditional update — either it affects one row (we won) or none (someone else holds it). There is no check-then-act window. Leases expire, so an instance killed mid-run does not hold the lock forever.

A job that throws is swallowed and logged. An unhandled rejection inside a cron tick would kill the API process that people are using to log their hours. Job status and a "run now" button live in Admin › Settings.

Refresh tokensOpaque, hashed at rest, rotated on use, family-revoked on replay (RFC 6819 §5.2.2.3)
In the browserThe access token lives in a module variable only — never localStorage, so XSS cannot read it. The refresh token is an httpOnly cookie the JS never sees.
Every requestThe user is re-read from the database. A stateless token cannot know the account was deactivated ninety seconds ago.
PaginationHard page-size ceiling (?pageSize=1000000 is a trivial DoS) and a sort-field allow-list — passing sortBy straight through lets a caller sort by passwordHash, which with pagination is a practical oracle for extracting hashes.
ErrorsOne exit point. Prisma errors are translated, never leaked — a raw Prisma error exposes table and column names. Unexpected errors return only a correlation ID.
UploadsAvatars verified by magic bytes, not by file extension
AuditAppend-only; sensitive fields dropped from diffs entirely rather than masked

10Operations

Deploying

DEPLOYMENT.md at the repo root carries the full procedure; docker-compose.yml, azure-setup.sh and azure-deploy.sh are the scaffolding. In production:

  • Use the restricted application user from db-setup.sql, not root.
  • Set TRUST_PROXY to the number of proxies in front of the app. Get it wrong and every audit row records the load balancer's IP, and every IP rate limit becomes one shared bucket for the whole internet.
  • Watch connection_limit: a pool per instance adds up fast against a small MySQL tier.
  • Run migrations with db:deploy. Never db:reset — it drops everything.

Backups and retention

Task history is deleted after TASK_RETENTION_DAYS (180). That is a business policy, so it is configurable — but it is a real delete. Confirm your database backup covers the retention horizon before going live, because after the sweep the only remaining record of a deleted period is the audit log.

Testing

SuiteCommand
Unitnpm run test:unit
Integrationnode --env-file=.env tests/integration/api.smoke.mjs
Data preservationnode --env-file=.env tests/integration/data-preservation.mjs
Run one integration suite per fresh database

These suites mutate shared state and are not idempotent. data-preservation.mjs deliberately deletes a seeded employee — that is the point of it — and api.smoke.mjs signs in as that same person. Running them back to back produces failures that look like product bugs and are not. Reset, restart the API, run one suite. Isolated, both are green.

Start the API with node --env-file=.env src/server.js for test runs, not npm run dev: the dev script uses --watch, which restarts mid-suite and shows up as ECONNRESET. Both suites honour SMOKE_ADMIN_EMAIL and SMOKE_ADMIN_PASSWORD.

11Known gaps

  • Rotate the credentials used during development. The Gemini API key and the Gmail app password were both exposed in working sessions and should be reissued before go-live. Add SMTP_REQUIRE_TLS=true.
  • Reuse of one password across MySQL and the application's admin account should be separated.
  • The AI's aiSummary / aiRiskScore fields on assignments are dormant seams, not features.
  • Notifications are polled (30–60s), not pushed. Fine at this scale; the seam for websockets is clean if needed.
