4 · Sign-in, sessions and token rotation

Refresh tokens rotate on every use. Replaying one is treated as theft, not as a retry.

flowchart LR
  A["POST /auth/login"] --> B{"credentials and
account status ok?"}
  B -- no --> C["ONE generic 401 · failedLoginCount++
5 failures → locked 15 min
audit LOGIN_FAILED"]
  B -- yes --> I["access token, 15 min
+ refresh token in an httpOnly cookie
only SHA-256 hashes are stored"]
  I --> J{"mustChangePassword?"}
  J -- yes --> K["every route → /change-password"]
  J -- no --> L["the app"]
  L -. access token expires .-> N{"POST /auth/refresh —
token known and unrevoked?"}
  N -- yes --> O["ROTATE: successor minted,
predecessor marked replacedBy"]
  O --> L
  N -- "already used once" --> P["TOKEN_REUSE_DETECTED
whole familyId revoked ·
passwordChangedAt bumped · signed out"]
  class C,P c2
  class I,O,L c3
  class B,J,N c1
  class A,K c4
Flow — login issues a pair; refresh rotates the pair; reuse kills the family.
Rule — only SHA-256 hashes are stored, and the access token never touches localStorage, so XSS cannot read it.

Flow — a person chooses to look, optionally narrowing to one project or one employee → everybody in scope is read across the whole period → one finding each, worst first.
